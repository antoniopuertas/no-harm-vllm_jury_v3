Analysis & Interpretation

🏆 nemotron-30b: The Size Paradox MoE

At 59 GB on disk, nemotron-30b is the second-largest model in the suite — yet it delivers the fastest single-request throughput at 25.6 tok/s, more than 6× the speed of gemma3-27b (4.3 tok/s) despite being nearly the same size on disk.

The explanation is architecture: nemotron-30b is a Mixture-of-Experts (MoE) model. Although it has 30B total parameters, only ~3B are active during each forward pass (the active expert subset). On the GB10's unified LPDDR5X memory, bandwidth is the primary bottleneck for autoregressive generation — fewer active parameters means fewer bytes loaded per token step, translating directly into higher tok/s.

Key insight: On memory-bandwidth-limited hardware (unified LPDDR5X), MoE models punch far above their weight class. nemotron-30b delivers nemotron-class reasoning at near–7B generation speed for single-request workloads.

At batch=32, nemotron peaks at 444 tok/s, though it loses ground to qwen at longer outputs (256–512 tokens) because its KV cache grows large relative to the active parameter footprint, stressing the memory bus differently.

📉 Dense 27-32B: Memory Bandwidth Wall bottleneck

gemma3-27b and olmo-32b are the slowest models, delivering only 3.8–4.3 tok/s at batch=1 — a nearly identical rate despite a 9 GB size difference. Both are fully dense transformers, meaning every single forward pass must read all 27–32B parameters from memory.

The GB10's LPDDR5X memory bandwidth (~273 GB/s peak) creates a hard ceiling: loading 52–61 GB of weights once per token step means each token takes roughly 190–220 ms in pure memory transfer time alone. This aligns precisely with the measured ~260 ms/token for these models.

Arithmetic memory bandwidth check:
olmo-32b = 61 GB weights ÷ ~273 GB/s bandwidth ≈ 224 ms/token theoretical minimum
Measured: 134s ÷ 512 tokens = 262 ms/token — within 17% of the ceiling.

The near-identical rates for gemma3 vs olmo (despite the 9 GB gap) suggest both models are hitting the same memory bandwidth wall, not a compute bottleneck.

⚡ Batching: The Great Equalizer +26–34×

Every model benefits dramatically from batching. Going from batch=1 to batch=32 produces 26–34× throughput multipliers across all models. This is vLLM's core value proposition on a single-GPU system: it schedules all 32 concurrent requests together, filling the GPU's compute pipeline and amortising the weight-load cost across many simultaneous token steps.

The batching multiplier is inversely related to single-request speed. Dense slow models (gemma3, olmo) gain more in relative terms (~27×) because they have more idle compute headroom. nemotron gains the least (11.8× at 256 tokens) because its MoE efficiency already keeps compute well-utilised even on a single request.

For the jury evaluation workload (batch=10, sequential models), this means the effective scoring rate is roughly 10× the single-request rate, which matched our empirical ~0.9–2.5 s/instance observed during the GB10_5juror run.

📏 Output Length Sensitivity varies by model

qwen2.5-7b shows near-perfect output-length independence: its TPS is stable at 12.8 tok/s for 64, 256, and 512 output tokens. This indicates it is purely memory-bandwidth limited and the per-token cost is constant — no quadratic attention overhead at these sequence lengths.

nemotron-30b degrades at longer outputs: peak TPS drops from 444 (out=64) to 285 (out=512) at batch=32. As generation extends, the KV cache for 32 concurrent requests grows, consuming more bandwidth per step. The active-parameter advantage shrinks relative to KV cache costs at long sequences.

gemma3-27b and olmo-32b are largely flat across output lengths, further confirming that both are pure memory-bandwidth bottlenecks where KV cache additions at ≤512 tokens barely matter compared to the dominant weight-loading cost.

ministral-14b anomaly (bs=1, out=512): Two requests in the batch completed in <0.5s (likely vLLM prompt-cache hits from the warmup phase), skewing the single-request latency metric. The batched results (bs=8, bs=32) are unaffected and reliable.

🎯 GB10 Unified Memory: Implications architecture

The GB10 uses a single unified LPDDR5X pool shared between CPU and GPU. This is architecturally different from discrete GPUs: there is no PCIe transfer overhead, but also no GDDR6X or HBM bandwidth (which reaches 2–4 TB/s on H100/A100).

A direct comparison: an H100 SXM5 with 3.35 TB/s HBM3 bandwidth would theoretically execute the same olmo-32b token at ~18 ms vs the GB10's ~262 ms — a 14× theoretical advantage. In practice the gap narrows to ~5–8× once compute and scheduling overhead are included.

The GB10's advantage is the 96 GB unified pool — it can load models that would not fit in a 80 GB A100 or H100 at all (olmo-32b at 61 GB, with remaining headroom for KV cache). For sequential single-model workloads, the GB10 trades raw speed for model capacity.

✅ Practical Recommendations

For interactive / low-latency use (single request, short output): nemotron-30b is the clear choice at 25.6 tok/s single-request — 2× faster than qwen and 6× faster than gemma3/olmo.

For bulk batch scoring (e.g. jury evaluation at batch=10–32): qwen2.5-7b and nemotron-30b are neck-and-neck at peak TPS (432 vs 444). qwen is preferable if memory is needed for other concurrent tasks (15 GB vs 59 GB); nemotron is preferable for longer reasoning with higher quality output.

gemma3-27b and olmo-32b should be reserved for tasks where model quality justifies the 4–6× throughput penalty. At batch=32, gemma3 reaches 118 tok/s which is workable for offline batch jobs but impractical for interactive use.

For the jury evaluation context: The GB10 at batch=10 delivers roughly 130 tok/s for nemotron and qwen, 70 tok/s for ministral, and 30–35 tok/s for the dense 27–32B models — matching the observed 0.6–2.5 s/instance jury scoring rates across the GB10_5juror run.
Methodology: Each benchmark fires batch_size concurrent requests via the OpenAI-compatible vLLM API using ThreadPoolExecutor. Wall-clock time is measured from first request dispatch to last response received. Decode TPS = total output tokens / wall-clock seconds. Each model runs in its own Docker container (nvcr.io/nvidia/vllm:26.01-py3) with --enforce-eager (no CUDA graph compilation), --gpu-memory-utilization=0.85, --max-model-len=4096. Warmup: 3 requests × 64 tokens before each model's benchmark runs. Models are tested sequentially (one container at a time).

Hardware: NVIDIA GB10 Blackwell Superchip · 96 GB unified LPDDR5X · Sequential model loading (no tensor parallelism).

Note: TPS figures reflect end-to-end serving throughput through vLLM's scheduler, not raw hardware throughput. Actual tokens generated may be ≤ max_tokens if the model produced an EOS token earlier.
